## 三、五维度详解

### 1. 技术栈健康度 — 38/50 ⭐⭐⭐⭐

**好在哪**
- 框架版本全部站在第一梯队：Next 16 + React 19 + Tailwind v4，充分利用了新能力（`optimizePackageImports`、`@tailwindcss/postcss`）。
- 依赖管理规范：`pnpm-lock.yaml` 存在，依赖可复现；`packageManager` 字段锁定 pnpm 版本，避免"在我机器上能跑"。
- ESLint 9 扁平配置 + `eslint-config-next`，并针对 react-hooks v7 做了精细化：启用 `set-state-in-effect`、`immutability` 为 error，`exhaustive-deps` 降为 warn——体现了对闭包与不可变性的约束意识。

**差在哪 / 风险**
- **无 TypeScript**：在 React 19 / Next 16 时代，纯 JS 对 `useFileConversion` 这类携带大量 ref/闭包/Worker 消息契约的逻辑是实打实的风险点（消息 `{type, percent, buffer, name}` 全部靠约定，无类型保护）。这是本维度最大扣分项。
- **依赖与注释不一致**：`next.config.mjs` 注释称"仅保留实际使用的 react-icons"，但 `react-lottie-player` 仍在 `Lottie.jsx` 中被动态引入并真实使用。注释误导，应修正。
- `react-lottie-player`（约 534KB 的 `welcome.json` 配套）对纯工具型应用是偏重的动画方案，值得评估是否可用 CSS/SVG 动画替代。

**改进建议**
- P0：至少为 `utils/`、`hooks/`、`workers/` 引入 TypeScript 或严格的 JSDoc + `// @ts-check`，优先覆盖 Worker 消息协议与 hook 返回值。
- 修正 `next.config.mjs` 注释，使其与依赖现状一致。

### 2. 架构设计模式 — 35/50 ⭐⭐⭐

**好在哪**
- 目录组织清晰：组件按文件夹拆分（`components/Xxx/Xxx.jsx`），业务逻辑下沉到 `hooks/`（useFileHandling / useFileConversion / useConversionManager / useFileDownloadManager），工具函数与 Worker 分离，职责基本单一。
- 状态/逻辑解耦做得不坏：转换主流程、下载管理、取消/暂停/恢复都封装在自定义 hook 里，且 `useFileConversion` 用 `filesRef`/`directionRef` 规避了闭包捕获过期值的经典坑。
- Web Worker 设计到位：重计算（EPUB 解析 + OpenCC 转换）放到 Worker，主线程降级路径完整，且对 Worker 崩溃 / 取消时的 **Promise 悬挂做了显式 reject 兜底**（`pendingRejectRef`、`onMessageError`），这是多数项目会遗漏的健壮性细节。

**差在哪 / 风险**
- **Home 上帝组件**：`src/pages/index.jsx` 持有约 15 个 `useState`/`useRef`，并把 ~20 个 props 下钻给 `UploadSection`/`ConvertedSection`（如 `isDragging / setIsDragging / isClicking / setIsClicking / isHoveringUpload…`）。这是典型的 prop drilling 反模式，组件耦合度高、单测困难。
- **无状态管理层**：全局状态散落在页面与 hook 之间，没有 Context 或轻量 store（如 Zustand）做状态收口。UI 状态（背景、方向、弹窗）与业务状态（文件、进度、结果）混在一起。
- 水合安全处理有亮点（`useSyncExternalStore` 用于 ThemeToggle、DirectionSelector 的 mounted 判定，sessionStorage 读取放到 `requestAnimationFrame`），但 `index.jsx` 里音频初始化、背景随机、错误态复位仍堆在页面层，关注点未充分分离。

**改进建议**
- P0：引入轻量状态库（Zustand）或 React Context，把"文件 / 转换 / 下载 / UI"四类状态分桶；`UploadSection` 改为消费 store 的容器组件，砍掉 prop 下钻。
- 将 `index.jsx` 中音频、背景、错误复位等副作用拆为独立 hook（如 `useCompletionSound`、`useRandomBackground`），页面只做编排。

### 3. 工程化成熟度 — 30/50 ⭐⭐⭐

**好在哪**
- 构建配置扎实：`compress`、`productionBrowserSourceMaps: false`、`async headers()` 对静态资源（动画 JSON / mp3 / ico / svg / woff2）设 `max-age=31536000, immutable`；动画 JSON 预加载用 `media="(min-width: 1024px)"` 按带宽条件化——这是有思考的优化。
- `next dev --turbopack` 提速开发体验；`optimizePackageImports: ['react-icons']` 减小图标体积。
- 有 Vitest 单测（虽少但真实存在），且 `jszip`/`zipUtils` 走动态 `import()` 做按需分包。

**差在哪 / 风险**
- **无 CI/CD**：仓库无 `.github/` 流水线，lint / test / build / 类型检查全靠人工本地跑，回归无人兜底。
- **无 Prettier、无 Husky / lint-staged**：只有 ESLint，代码风格无法自动统一，提交前无自动校验门禁。
- **类型检查门禁缺失**：因无 TS，少了一道编译期防线。
- `pnpm-workspace.yaml` 在单包场景下是"提前量"而非必需，反而增加认知负担（除非已有多包规划）。
- 无 bundle 分析（`@next/bundle-analyzer`），无法量化 `framer-motion` / `react-lottie-player` 的体积贡献。

**改进建议**
- P1：补 Prettier + Husky + lint-staged（提交时自动格式化 + lint），并加一个最小 GitHub Actions：install → lint → test → build。
- 评估 `pnpm-workspace.yaml` 是否必要；接入 `@next/bundle-analyzer` 设定体积基线。

### 4. 性能与可维护性 — 41/50 ⭐⭐⭐⭐

**好在哪（性能侧很强）**
- 真正的重活（EPUB 解压 + 逐文件 OpenCC 转换）在 **Web Worker** 执行，主线程不冻结，大文件体验有保障。
- 动态导入控制体积：`jszip`（打包下载）、`zipUtils`（主线程降级路径）均按需 `import()`。
- 静态资源长期缓存 + 首屏骨架由 CSS 即时绘制，LCP 不被大 JSON 阻塞；大屏才预加载欢迎动画——带宽考虑周到。
- 暂停/恢复使用 `resumeResolverRef` 挂起主循环而非轮询，取消用 `AbortController` + Worker `terminate()` 双保险，资源清理（`resetConversion` 兜底 terminate）严谨。

**差在哪（可维护性侧）**
- 大组件导致的可维护性下滑（同维度 2）：`index.jsx` 288 行、巨型 props 列表，任何改动都需通读整页。
- 测试环境为 `node`（`vitest.config.js` `environment: "node"`），**无法测试任何依赖 DOM 的组件**，目前仅 2 个纯函数测试（opencc、zipUtils，共 146 行），组件 / 集成 / E2E 全缺，覆盖率极低。
- `globals.css` 533 行且 `tailwind.config.js` 的 `safelist` 较长（约 50 条渐变/动画类），一旦 purge 策略变动易踩坑，建议审视是否可由动态类名改为内联/数据驱动。

**改进建议**
- P1：Vitest 环境切到 `jsdom` + `@testing-library/react`，优先补"选择文件→转换→下载"核心链路与 Worker 降级路径的集成测试，并设覆盖率阈值（如 utils/hooks ≥ 70%）。
- P2：梳理 `safelist`，将随机背景渐变改为数据驱动（按 scheme key 映射有限类集合），减少维护面。

## 五、结论

epub-zh-convert 是一个**技术选型先进、运行期性能优秀、关键健壮性细节到位**（Worker 卸载、Promise 悬挂兜底、水合安全）的工具型前端项目，基本功明显好于平均水平。主要短板集中在**工程化闭环**（无 CI / 无格式化门禁 / 测试薄弱）与**架构收口**（上帝组件、prop drilling、无状态管理层、缺 TypeScript）。

按优先级补齐 P0（TS + 状态收口）与 P1（CI + 测试），项目可从"能跑的好项目"升级为"可持续维护的标杆项目"。架构没有银弹，合适的才是最好的——以上建议结合团队规模与迭代节奏取舍即可。
